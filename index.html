About

I'm an MEng Electronic & Information Engineering student at Imperial College London. Most of my interest sits at the boundary between software and hardware: register-transfer-level design, processor microarchitecture, embedded systems, and accelerating real workloads on FPGAs.

I like problems where the constraint is the design — a fixed clock, limited DSP slices, a video frame that has to be ready before the next pixel-clock edge. Recently that's meant building a fully pipelined neural network on a PYNQ-Z1 that renders a classifier's decision boundary in real time, about 40× faster than the CPU doing the same maths.

Alongside the engineering, I've spent time in very different rooms — backend development at Accenture, a trading academy at Optiver, and leadership in the Singapore Armed Forces. Currently open to internship opportunities in hardware, FPGA and embedded engineering.

PROJ·01

Hardware-Accelerated Neural Network Decision Visualiser

RTL / hardware accelerator

A fully pipelined 16→64→32→3 MLP built in SystemVerilog on a PYNQ-Z1, running one complete neural-network forward pass per pixel to render a classifier's decision boundary as a live colour heatmap — 640×480 streamed to HDMI at roughly 20 fps. That's about 40× faster than an equivalent C++/SDL2 CPU baseline (and, by estimate, ~730× more energy-efficient), saturating all 220 DSP slices on the chip. The same hardware retargets any tabular dataset of up to 16 features at runtime; demoed on Iris and the 16-feature Dry Bean set. I led the core HDL datapath with one teammate — the multiply-accumulate units, the three-layer pipeline, weight RAM, and the output FIFO.

PROJ·03

5-Stage Pipelined RISC-V CPU + Data Cache

RTL / microarchitecture

An RV32I processor built from scratch in SystemVerilog: a full five-stage pipeline with forwarding, load-use stalls, and branch/jump flushing, paired with a 2-way set-associative write-back data cache (LRU, multi-cycle miss handling). Validated with custom Verilator testbenches, including a Gaussian-histogram workload built specifically to stress hazards and cache behaviour. The team also pushed into superscalar and branch-prediction variants.

PROJ·05

Multithreaded UDP Chat System

systems / concurrency

A concurrent UDP client–server chat written in C. Each client runs dedicated sender and listener threads while the server spawns a thread per packet, all coordinated through a thread-safe client registry using reader–writer locks. Supports broadcast and private messaging, mute, rename and admin kick, with a 15-message history buffer and a min-heap inactivity-removal engine driven by ping/pong logic.
